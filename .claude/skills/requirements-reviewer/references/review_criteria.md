# Review Criteria

## 評価項目と基準

### 1. 部門の網羅性 (Completeness of Departments)

| 基準 | 合格 | 不合格 |
|---|---|---|
| 全関連部門が列挙されている | ✅ | |
| 各部門の役割が記述されている | ✅ | |
| 暗黙の関与者がいない | ✅ | |
| 部門間の上下関係が推測可能 | ✅ | |

### 2. ハンドオフの明示性 (Handoff Clarity)

部門間でタスクが受け渡される全ての箇所で:
- 受け渡し方法（メール、システム通知、口頭等）が記載されているか
- 受け渡しのトリガー条件が明確か
- 待ち時間や SLA の記載があるか（あれば望ましい）

### 3. 判断ポイントの完全性 (Decision Completeness)

各判断ポイントで:
- 判断条件が定量的に記述されているか（例: 「差異5%以上」vs「差異がある」）
- Yes/No 両方の分岐先が明示されているか
- 判断の担当者が明確か
- 例外的な判断結果（保留、エスカレーション等）が考慮されているか

### 4. 工程フェーズの適切性 (Phase Granularity)

- フェーズが時間軸または論理的順序で並んでいるか
- フェーズの粒度がカラム数 10-20 の範囲に収まるか
- 各フェーズに少なくとも1つのタスクが割り当て可能か
- フェーズ名が明確で重複がないか

### 5. システム・ツールの特定 (System Identification)

- 各タスクで使用するシステムが明記されているか
- システム名が正式名称で統一されているか
- システム間のデータ連携が示唆されているか

### 6. 例外パスの文書化 (Exception Documentation)

- 正常フロー以外のパスが列挙されているか
- 差戻しフローの条件と戻り先が明確か
- エスカレーション条件が定義されているか
- タイムアウト処理が考慮されているか

### 7. 前提の妥当性 (Assumption Validity)

- (Assumption) タグ付きの前提が妥当か
- 前提同士に矛盾がないか
- 危険な前提（検証が必要な前提）が識別されているか

## 重要度の判定基準

### Critical
- チャート化に必要な基本情報の欠落
- 例: 主要部門が未特定、プロセスの開始/終了が不明

### Major
- チャートの正確性に影響する情報の欠落
- 例: 判断ポイントの片方の分岐先が不明、重要なハンドオフの記載漏れ

### Minor
- チャートの品質改善に寄与する修正
- 例: ラベル名の不統一、システム名の省略

### Info
- ベストプラクティスの提案
- 例: 「SLA を記載すると理解しやすくなる」
